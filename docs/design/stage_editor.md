# ステージエディタ 要件定義書

ブラウザ上で動作する3Dレベルエディタ。地面・壁などの配置物と敵を自由に配置し、
`stageData.json` として書き出す。書き出したJSONをゲーム本体（DxLib）が読み込んでステージを構築する。

対象ステージは **1つのみ**。複数ステージの管理・切り替えはスコープ外とする。

---

## 1. 目的

| | |
|---|---|
| 解決したい課題 | ステージのレイアウトを手書きJSONで作るのは非現実的。座標を目で見ながら配置したい |
| 成果物 | ① ブラウザ製エディタ（`tools/stage-editor/`）　② 新しい `stageData.json` スキーマ　③ それを読むゲーム側の実装 |
| 操作感の目標 | **Unityのシーンビュー同等**（フリーカメラ・移動/回転/スケールギズモ・Hierarchy・Inspector・Undo/Redo） |

## 2. ステージのコンセプト

### 2-1. 設定

プレイヤーは自分のWindows内部を進み、`C:\Program Files\Apple\` に居座るMacBookを排除する。

> Windowsに iTunes や Safari をインストールすると、実際に `C:\Program Files\` 配下に
> `Apple` フォルダが作られる。**自分のPCの中に正規にインストールされたApple領域が敵の巣窟**という構図。
> 道中の敵が Safari / Xcode である理由もこれで説明がつく。

### 2-2. 区画構成（一本道・3区画）

```
① C:\Users\Player\Desktop   … 導入。雑魚1〜2体
        ↓ 坂を下る（パス通路）
② C:\Program Files\          … 主戦場。ここの雑魚を全滅させるとボスが出現
        ↓ 坂を下る（青 → 白銀へ色調が反転）
③ C:\Program Files\Apple\    … ボスアリーナ。MacBook戦
```

雑魚全滅 → ボス出現 → 撃破でクリア、という [todo.md](../memo/todo.md) の「6. ボス戦フロー」にそのまま乗る。

### 2-3. ビジュアル指針

**世界は真っ黒な虚無で、床だけが浮いている。** 背景を作り込む必要がないため制作コストが低い。

| 用途 | 色 |
|---|---|
| 背景・虚無 | `#000000` 〜 `#0A0E14` |
| Windows側（自陣）の発光 | `#00A4EF` |
| フォルダの黄（挿し色） | `#FFC83D` |
| Apple領域（敵陣） | `#E8E8ED` / `#F5F5F7`（清潔で無機質な白銀） |
| 危険・ボス | `#E81123` |

道中の「暗い黒＋青のネオン」から、ボスアリーナで「眩しい白銀」へ切り替わる**明暗の反転**でボス戦の特別感を出す。

> **注意**：DxLibには標準でブルーム（発光のにじみ）がない。ネオン感は
> **テクスチャ側で発光しているように描く**のを基本とする。
> どうしても足りない場合のみ `GraphFilter` の `GRAPH_FILTER_GAUSS` による簡易ブルームを検討する。

### 2-4. 配置物の種類（初期カタログ）

すべて**立方体をXYZに引き伸ばしたもの**で表現できる。用意が必要なのは主にテクスチャ。

| 種類ID | 表示名 | 見た目 |
|---|---|---|
| `floor_folder` | フォルダ床 | 平たい板。エクスプローラーの詳細表示のような等間隔の水平線。縁が青く発光 |
| `path_corridor` | パス通路 | 細長い板。表面を `C:\Users\Player\...` の文字が流れる。**傾けて坂に使う** |
| `wall_window` | ウィンドウ壁 | 巨大なエクスプローラーのウィンドウ枠。上端にタイトルバー、右上に − □ ✕ |
| `pillar_folder` | フォルダ柱 | 黄色い折れたフォルダ形の柱。戦闘中の遮蔽物 |
| `block_file` | ファイルブロック | `.exe` `.dll` `.txt` アイコン付きの立方体。積んで障害物に |
| `floor_apple` | アリーナ床 | 白銀・つや消し・継ぎ目のない滑らかな面 |

## 3. 座標系と単位

### 3-1. 単位

ゲーム内の既存データから逆算した基準値。

| 基準 | 値 |
|---|---|
| **1ユニット** | 約 1cm |
| プレイヤーの身長（コライダー） | **150** |
| プレイヤーの攻撃リーチ | 200 |
| safariの索敵範囲 | 1200 |
| safariの間合い | 400〜600 |

| 設計値 | 推奨サイズ |
|---|---|
| 1回の戦闘が成立する空間 | 直径 2000〜2500 |
| 通路の幅 | 800〜1200 |
| ボスアリーナ | 直径 3000 前後 |
| ステージ全体 | 4000〜6000 四方 |
| **グリッド単位** | **100** |

### 3-2. 座標系（重要）

**JSONに保存する座標は DxLib の座標系（左手系・Y-up）とする。** 変換はエディタ側で吸収する。

three.js は右手系のため、そのまま表示するとZ方向が反転する。対策：

> **推奨**：three.js のシーンに `scale.z = -1` を掛けたルートコンテナを1つ置き、
> 配置物はすべてその子にする。子の `position` / `rotation` に **DxLibの値をそのまま入れられる**ようになり、
> ギズモのドラッグ方向もDxLibの符号と一致する。
>
> 副作用：面の裏表が反転するため、マテリアルは `side: THREE.DoubleSide` を指定すること。
> ライティングの陰影がわずかにずれるが、本エディタは形状確認が目的なので許容する。

**検証必須**：エディタで `position: [0, 0, 800]` に置いた敵が、ゲーム内でも現在の `mac` と同じ位置に出ることを最初に確認する。

### 3-3. 回転

`rotation` は `[x, y, z]` の3値・**度数法**で保持する。

- 通常の配置物は Y のみ使う（水平に回す）
- **坂は X または Z を傾けて作る**
- 既存の [groundData.json](../../assets/data/groundData.json) が `rotX/rotY/rotZ` を持っているため、データ構造の前例あり

## 4. データ仕様

### 4-1. `assets/data/stageCatalog.json`（新規）

配置物の種類定義。**エディタとゲームの両方が読む**。エディタは選択肢とプレビュー色を、
ゲームは種類ID → モデルの解決に使う。

```jsonc
{
  "props": [
    {
      "id": "floor_folder",
      "displayName": "フォルダ床",
      "model": "assets/model/stage/FloorFolder.mv1",
      "baseSize": [100, 100, 100],   // モデル素材の実寸。size ÷ baseSize がモデルスケールになる
      "editorColor": "#00A4EF",      // エディタでの箱の色
      "collider": "box",             // "box" | "none"
      "defaultSize": [1000, 20, 1000]
    }
  ],
  "enemies": [
    { "id": "safari", "displayName": "Safari", "editorColor": "#4DA6FF", "editorSize": [200, 300, 200] },
    { "id": "xcode",  "displayName": "Xcode",  "editorColor": "#1E90FF", "editorSize": [200, 400, 200] }
  ],
  "boss": { "id": "mac", "displayName": "MacBook", "editorColor": "#E8E8ED", "editorSize": [400, 600, 400] }
}
```

> **運用上の注意**：`enemies` の `id` は [assets/data/enemies/](../../assets/data/enemies/) の各JSONの
> `id` と対応していなければならない。敵を追加したときは**2箇所の更新が必要**。
> ゲーム起動時に不一致を検出したら警告ログを出すこと。

### 4-2. `assets/data/stageData.json`（拡張）

```jsonc
{
  "playerStart": {
    "position": [0, 0, -1500],
    "rotationY": 0
  },
  "props": [
    {
      "type": "floor_folder",
      "position": [0, -50, 0],
      "rotation": [0, 0, 0],
      "size": [3000, 20, 3000]
    },
    {
      "type": "path_corridor",
      "position": [0, -200, 2000],
      "rotation": [-15, 0, 0],      // X軸を傾けて坂にする
      "size": [1000, 20, 2000]
    }
  ],
  "spawns": [
    { "type": "safari", "position": [-200, 0, 150], "rotationY": 0 }
  ],
  "mac": {
    "type": "mac",
    "position": [0, 0, 800],
    "rotationY": 180
  }
}
```

### 4-3. フィールド定義

| キー | 型 | 必須 | 説明 |
|---|---|---|---|
| `playerStart.position` | `[x,y,z]` | ○ | プレイヤーの開始位置 |
| `playerStart.rotationY` | `number` | ○ | 開始時の向き（度） |
| `props[].type` | `string` | ○ | `stageCatalog.props[].id` を参照 |
| `props[].position` | `[x,y,z]` | ○ | 配置物の中心座標 |
| `props[].rotation` | `[x,y,z]` | ○ | 度数法。坂はX/Zを使う |
| `props[].size` | `[x,y,z]` | ○ | **実寸（ユニット）**。モデルスケールではない |
| `spawns[].type` | `string` | ○ | `stageCatalog.enemies[].id` を参照 |
| `spawns[].position` | `[x,y,z]` | ○ | スポーン位置 |
| `spawns[].rotationY` | `number` | ○ | 初期の向き（度） |
| `mac` | `object` | ○ | **ボスは `spawns[]` に統合せず別枠のまま**（既存仕様を維持） |

### 4-4. 設計上の決定事項と理由

| 決定 | 理由 |
|---|---|
| **`size` は実寸で持つ**（モデルスケールではない） | エディタで「3000」と入力すれば3000になる。スケール倍率だとモデルの素材サイズを意識させられる |
| **スケールはXYZ個別**（一様1値ではない） | 立方体1モデルを引き伸ばして床・通路・壁・柱をすべて賄える（グレーボックス方式）。用意するモデルを激減できる |
| **敵にスケールを持たせない** | 「巨大safari」が作れてしまうと当たり判定・攻撃レンジとの整合が崩れる。サイズは `enemies/*.json` 側の責務 |
| **当たり判定は種類（カタログ）側に持たせる** | 同じモデルなら判定も同じ。配置ごとに指定させると入力が増えるだけ |
| **ボスは `mac` 別枠のまま** | 既存の [StageRepository](../../src/infrastructure/resource/repository/StageRepository.cpp) の読み込みと、雑魚全滅判定の対象から自然に外れる構造を維持するため |

## 5. エディタ機能要件

### 5-1. 配置場所・技術構成

| | |
|---|---|
| 配置 | `tools/stage-editor/` |
| 形態 | ブラウザで開く静的HTML。ビルド不要 |
| 3D | three.js（`TransformControls` を使用。カメラは自前のフライスルー実装） |
| 命名規則 | [javascript_naming_convention.md](../conventions/javascript_naming_convention.md) に従う |

> ⚠️ **ES Modules と importmap を使うため `file://` では動かない。**
> `python -m http.server` などのローカルサーバー経由で開く必要がある。
> 起動手順は [tools/stage-editor/README_ja.md](../../tools/stage-editor/README_ja.md) を参照。

### 5-2. 画面構成

```
┌──────────────────────────────────────────────────────┐
│ ツールバー： [移動][回転][拡縮] │ [Local/Global] │ [スナップ] │ [保存][読込] │
├────────────┬────────────────────────────┬────────────┤
│ Hierarchy  │                            │ Inspector  │
│            │                            │            │
│ ▼ Props    │        3Dビューポート        │ Type: ...  │
│   Floor_01 │                            │ Pos  X Y Z │
│   Wall_02  │      （グリッド＋軸表示）      │ Rot  X Y Z │
│ ▼ Enemies  │                            │ Size X Y Z │
│   safari_1 │                            │            │
│ ▼ Boss     │                            │            │
│ ▼ Player   │                            │            │
├────────────┴────────────────────────────┴────────────┤
│ パレット： [フォルダ床][パス通路][ウィンドウ壁][柱][ファイル][敵…] │
└──────────────────────────────────────────────────────┘
```

| パネル | 要件 |
|---|---|
| **3Dビューポート** | 配置物を色付きの箱として表示。グリッド（100単位）と原点の軸を表示 |
| **Hierarchy** | 配置物・敵・ボス・プレイヤー開始位置をカテゴリ別に一覧。クリックで選択、選択中はハイライト |
| **Inspector** | 選択中オブジェクトの `type`（表示のみ）・位置・回転・サイズを数値入力で編集。ギズモ操作と双方向に同期 |
| **パレット** | `stageCatalog.json` から動的に生成。クリック → ビューポートの中心付近に `defaultSize` で新規配置 |

> モデルの見た目は**色付きの箱で十分**とする。何の種類かが判別できればよく、
> `.mv1` はブラウザで読めないため実物表示は行わない。箱には種類の表示名をラベルとして重ねる。

### 5-3. カメラ操作（Unityのシーンビュー準拠）

| 操作 | 動作 |
|---|---|
| 右ドラッグ | 視点回転（look around） |
| 右ドラッグ中の `WASD` | 前後左右へ移動（フライスルー） |
| 右ドラッグ中の `Q` / `E` | 下降 / 上昇 |
| 右ドラッグ中の `Shift` | 移動速度アップ |
| 中ドラッグ | 平行移動（パン） |
| ホイール | ズーム |
| `F` | 選択中オブジェクトにフォーカス（カメラを寄せる） |
| 左ドラッグ（オブジェクト外） | 何もしない（誤操作防止） |

### 5-4. オブジェクト操作

| 操作 | 動作 |
|---|---|
| 左クリック | オブジェクトを選択 |
| 左クリック（何もない場所） | 選択解除 |
| ギズモをドラッグ | 移動 / 回転 / 拡縮（ツールバーのモードに従う） |
| `W` / `E` / `R` | 移動 / 回転 / 拡縮モードへ切り替え |
| `Ctrl` を押しながらドラッグ | **グリッドスナップ**（移動=100単位／回転=15度／拡縮=100単位） |
| `Ctrl + D` | 選択中を複製（少しずらして配置） |
| `Delete` | 選択中を削除 |
| `Ctrl + Z` / `Ctrl + Y` | Undo / Redo |
| Local / Global 切り替え | ギズモの軸をオブジェクトのローカル軸／ワールド軸で切り替え（坂の編集に必要） |

**Undo/Redo の対象**：追加・削除・複製・移動・回転・拡縮・Inspectorでの数値変更。
履歴は50手程度を保持する。

### 5-5. 入出力

| 機能 | 要件 |
|---|---|
| **読み込み（必須）** | 既存の `stageData.json` をファイル選択で読み込み、編集を再開できる。**これが無いと一度保存したステージを二度と編集できない** |
| **カタログ読み込み** | 起動時に `stageCatalog.json` を読み込んでパレットを構築。ファイル選択で差し替え可能 |
| **書き出し** | `stageData.json` としてダウンロード。ブラウザから `assets/data/` へ直接書き込むことはできないため、**ユーザーが手動で配置する** |
| **自動保存** | 編集内容を `localStorage` に自動保存し、リロードしても失われないようにする |
| **バリデーション** | 書き出し前に、カタログに存在しない `type`・サイズ0以下・プレイヤー開始位置未設定を検出して警告 |

### 5-6. スコープ外（今回は作らない）

- 複数選択、コピー＆ペースト
- 複数ステージの管理・切り替え
- `.mv1` モデルの実物プレビュー
- ライティング・マテリアルの編集
- ゲーム本体との連携（ホットリロード等）
- 落下・穴の配置（**床は繋がっている前提。落下処理は仕様から除外**）

## 6. ゲーム側に必要な変更

エディタ単体では完結しない。以下はゲーム本体（C++）側の作業。

### 6-1. 配置物の複数化 🔴 必須

現在の [Ground](../../src/game/stage/Ground.h) は **Entity 1つの単数前提**、
[groundData.json](../../assets/data/groundData.json) も単一オブジェクト。これを配列化する。

- `groundData.json` を廃止し、`stageData.json` の `props[]` へ統合
- `Ground` クラスを廃止、または汎用の `StageProp` へ置換して複数生成に対応
- [StageMetadata](../../src/core/data/StageMetadata.h) に `PropMetadata`（type / position / rotation / size）を追加
- [StageRepository](../../src/infrastructure/resource/repository/StageRepository.cpp) を新スキーマに対応
- `stageCatalog.json` を読む `StageCatalogRepository` を追加し、type → モデルパスを解決

### 6-2. 接地処理のレイキャスト化 🔴 必須（坂の前提条件）

**坂を入れる以上、これは避けられない。** 現在の当たり判定はAABB（軸に沿った箱）で、
箱を傾けた瞬間に表現できなくなる。

> **方式**：プレイヤーの足元から真下へレイを飛ばし、当たった面のY座標に乗せる。
> 坂・段差が1つの仕組みで解決する3Dアクションの標準手法。
> DxLibの `MV1SetupCollInfo` / `MV1CollCheck_Line` を使う案が
> [todo.md](../memo/todo.md) の「8. ステージ制作」に既に挙がっている。

想定工数：1〜2日。[schedule.md](../management/schedule.md) のフェーズB（7/27「配置物のコライダー」）に組み込む。

### 6-3. その他

| 項目 | 内容 |
|---|---|
| `size` → モデルスケール変換 | `modelScale = size ÷ catalog.baseSize` |
| コライダーのスケール適用 | **忘れやすい**。見た目だけ伸びて判定が元サイズのままになる事故に注意 |
| プレイヤー開始位置 | `playerStart` を読んで初期配置に反映 |
| 敵の初期向き | `spawns[].rotationY` を反映 |
| ステージ外周 | 落下しないよう壁で囲む（またはXZ範囲の制限） |

## 7. 実装順序の提案

エディタとゲーム側を並行させず、**「JSONが正しく往復すること」を先に確認する**順序にする。

| # | 内容 | 完了条件 |
|---|---|---|
| 1 | `stageCatalog.json` / 新 `stageData.json` のスキーマ確定と手書きサンプル作成 | サンプルJSONが存在する |
| 2 | ゲーム側：`props[]` の読み込み・複数配置物の生成と描画（坂はまだ無し） | 手書きJSONの床がゲーム内に出る |
| 3 | エディタ：ビューポート＋パレット＋配置＋書き出し | 書き出したJSONがステップ2で読める |
| 4 | エディタ：選択・ギズモ・Inspector・Hierarchy | Unity相当の操作ができる |
| 5 | エディタ：Undo/Redo・複製・削除・スナップ・読み込み | 実用に足る |
| 6 | ゲーム側：接地のレイキャスト化 | 坂を登り降りできる |
| 7 | ステージ本制作（3区画のレイアウト） | 通しプレイできる |

**ステップ3の完了時点で座標系の検証**（3-2参照）を必ず行うこと。ここを後回しにすると、
作り込んだステージが丸ごとZ反転している事故が起きる。
